# Hermes-on-Blaxel - project context

Self-hosted **Hermes Agent** (Nous Research) running in a **Blaxel cloud sandbox**, reachable as a
**Telegram bot**, backed by **any model provider** Hermes supports (default: GLM-5.1 via Z.AI), with an optional **web dashboard**. Texting the bot wakes
the sandbox; it sleeps when idle (cheap). The repo is meant to be **clone → `./scripts/setup.ps1` → live**.

## Repo rules (for Claude and contributors) - READ FIRST
- **Do NOT add Claude / any AI as a commit author or `Co-Authored-By:` trailer.** Commits are authored by the human only; keep commit messages clean.
- **Never commit real secrets OR personal identifiers** - API keys, tokens, the dashboard password, your Telegram id, the Blaxel **workspace name**, live preview URLs, or emails. They belong only in the gitignored `.env`; use placeholders (`<workspace>`, `<your-id>`) in docs and examples.
- **No em-dashes or en-dashes** in any file (docs, code, comments, commit messages). Use a plain hyphen (-), a comma, or reword.

## Quick orientation for Claude
- **Secrets/config live in `.env`** (gitignored). Never print real secret values into committed files. `.env.example` is the template.
- **All ops go through the Blaxel CLI** at `%LOCALAPPDATA%\blaxel\bl.exe` - call by full path: `& "$env:LOCALAPPDATA\blaxel\bl.exe"`. There is no SSH; you manage the box by running commands through its **process API** (`--path /process`).
- **Don't hand-run the long deploy one-liner.** Use `./scripts/deploy.ps1` (reads `.env`, deploys, re-binds both previews).
- **A full deploy WIPES runtime data** (lesson #10) → `./scripts/backup-data.ps1` before, `./scripts/restore-data.ps1` after.

## Architecture (one line)
Telegram msg → Telegram servers (retry on failure) → Blaxel edge/preview (wakes box) → **Hermes gateway :9099** (auth-checks the user, calls GLM) → reply.

## Deployment modes (`DEPLOY_MODE` in `.env`)
The gateway/dashboard are launched via the Blaxel **process API**, which has two relevant knobs the entrypoint sets:
- **`timeout`** - default **600 s (10 min)**, after which the process is **killed**. We always pass **`timeout: 0`** (indefinite) - without it the gateway/dashboard die every 10 min (the supervision loop relaunches them, but that's a ~10-15 s outage each cycle).
- **`keepAlive`** - `true` **prevents Blaxel standby** (always-on); `false` lets the box suspend when idle.

| `DEPLOY_MODE` | keepAlive | Behavior | Cost |
|---|---|---|---|
| `always-on` (default) | `true` | Box runs 24/7, instant replies, no outage cycle | continuous |
| `scale-to-zero` | `false` | Box suspends after ~15 min idle, wakes on a message (~1 min cold start) | ~zero when idle |

The supervision loop is the safety net in both modes (relaunches on resume in scale-to-zero, or after any crash in always-on).

## How to operate (cheat-sheet)
- **Status:** `bl get sandbox hermes-box`
- **Run a command in the box:** `bl run sandbox hermes-box --path /process --data '{"command":"...","waitForCompletion":true}'` (base64-encode complex commands; large payloads via `--file`).
- **Read a file:** `bl run sandbox hermes-box --method GET --path "/filesystem//root/.hermes/logs/agent.log"`
- **Logs (in box):** `/root/.hermes/logs/agent.log`, `gateway.log`, `gw-boot.log`
- **Webhook health (from PC):** `Invoke-RestMethod "https://api.telegram.org/bot<token>/getWebhookInfo"` - want `last_error_message: null`.
- **Redeploy:** `./scripts/deploy.ps1` (backup first if data matters). Config-only change: `./scripts/deploy.ps1 -SkipBuild`.
- **Verify:** telegram preview → HTTP 405; dashboard → 302 (→ login).

## Hard-won lessons (why it's built this way)
1. Blaxel's default image is Alpine/musl; Hermes needs glibc → **`node:22-slim`** (Debian).
2. Free tier: 4 GB RAM, **2 GB overlay, 0 volumes**. Full `[all]` install won't fit → lean install (`anthropic,web` + telegram), Hermes **baked into the image** (read-only layer).
3. **Webhook mode**, not polling (polling keeps the box awake 24/7).
4. Port **8443 is NOT preview-routable** → use **9099**.
5. Ports are only edge-routable if the process is launched via the **process API** (`curl localhost:8080/process`), not directly - so the entrypoint launches gateway+dashboard that way.
6. Stale gateway lock blocked restarts → lock dir on tmpfs `/dev/shm/hglock`, cleared before each launch.
7. **Process API kills launched processes on a default 600s (10-min) timer** (`exitCode -1`/SIGKILL) - *this* was the "gateway dies every 10 min" cycle, not suspend. **Fix: pass `timeout: 0`** in the launch (indefinite). Separately, in scale-to-zero mode suspend also kills them on resume. Either way the entrypoint's **supervision loop** (PID 1, survives both) relaunches gateway/dashboard when their port is unbound (~5-15 s). *Caveat:* the first message after a scale-to-zero sleep 502s once while waking → Telegram retries → reply within ~1 min.
8. **Provider-agnostic model config.** The entrypoint passes through ANY provider credential env var (`*_API_KEY`/`*_BASE_URL`/`*_AUTH_TOKEN`) and writes `config.yaml`'s model section from `MODEL_PROVIDER`/`MODEL_NAME`/`MODEL_BASE_URL`. So the template isn't tied to Z.AI - pick any provider in the wizard, or "configure later" and set it via the **dashboard** or **`bl connect … → hermes setup model`** (the two provider-setup paths). *Z.AI note:* its key works on both `/paas/v4` and `/coding/paas/v4`; Hermes caches the first-probed endpoint in `auth.json` (wins over config on a non-fresh box), so set the coding endpoint (`MODEL_BASE_URL`) **before first run**. Verify: `grep base_url /root/.hermes/auth.json`.
9. **Dashboard `--insecure` BYPASSES the login gate.** For username/password (Hermes `dashboard_auth/basic`), run **without** `--insecure` (still binds 0.0.0.0 when an auth provider is set) and make the preview **public** - the password is the lock. The entrypoint switches automatically on whether `HERMES_DASHBOARD_BASIC_AUTH_USERNAME` is set. Login: `POST /auth/password-login {provider:"basic",username,password}`; it's rate-limited.
10. **No persistent volume → every full `bl deploy` rebuild WIPES `/root/.hermes`** (sessions, memories, kanban). Sleep/wake does NOT wipe (same overlay) - only rebuilds do. Use the backup/restore scripts around rebuilds.
11. **Standby is controlled by `keepAlive`, and counts only INCOMING (edge) connections** - outbound links (gateway→Telegram) do NOT keep the box awake; loopback (the supervision loop's nc) doesn't either. With `keepAlive:true` (always-on) Blaxel never suspends; with `keepAlive:false` it suspends ~15 min after the last incoming connection. Standby is **not billed for memory** (only snapshot storage). See "Deployment modes".
12. **`bl deploy` must run from inside `hermes-box/`** - `-d` from the parent uploads an empty build context (`COPY entrypoint.sh: not found`). `scripts/deploy.ps1` handles this.
